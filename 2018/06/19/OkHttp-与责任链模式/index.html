OkHttp 与责任链模式

2018-06-19

大学上 UML 课程的时候总觉得设计模式遥不可及、无法理解，印象最深的是 UML 课程期末考试的最后一题是责任链模式相关的内容，我没有做出来。毕业后，也曾抱着《设计模式·可复用面向对象软件的基础》从头开始看，总感觉一头雾水，少点什么。

如今看来，当时还是见得太少，经验不足。

书中关于的责任链模式的主要例子是多视窗对用户事件的处理，Android 系统中 View 树对触摸事件的分发处理流程也用到了责任链模式。

该模式的核心想法是:

给多个对象处理一个请求的机会，从而解耦发送者和接受者，该请求沿着对象链传递直到其中一个对象处理它。

对于 Http 客户端请求，两个主要工作是处理缓存和网络请求，这两部分都有机会处理 Http 请求，而且存在一种场景是离线状态且缓存可用，这时的请求便只由缓存部分处理。采用责任链模式可以很优雅地将缓存和网络职责分离，而且 Cache 部分总是先于 Socket 处理请求。

每个 Interceptor 里的 intercept 函数会拦截请求，做自己的工作，然后调用 Chain 里的 proceed 函数，将 index 增 1，传递 Request 给下一个拦截器处理。

CallServerInterceptor 是个例外，由于是整条处理链的最后一个拦截器，这里直接处理网络请求，返回 Response，不再继续传递。

值得一提的是 CacheInterceptor 类对 Http 协议的缓存设计做了完整的实现，很适合拿来学习Http协议的缓存处理，具体的细节便不在这里细说了。

小结

OkHttp 源码读起来确实很优雅，很顺畅，但是如果是我们面对Http客户端的实现需求，是否能想到这么优雅的设计方案。和理解原理比起来，仅仅会使用库还远远不够；和设计实现比起来，仅仅理解原理也同样远远不够。
